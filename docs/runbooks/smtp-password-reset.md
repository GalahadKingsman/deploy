# Сброс пароля: SMTP и переменные API

Публичный поток: пользователь в модалке на лендинге → `POST /auth/password/reset/request` → письмо со ссылкой `PUBLIC_WEB_ORIGIN/reset-password?token=...` → подтверждение через существующие `GET .../preview` и `POST .../confirm`.

## Обязательные переменные (API)

| Переменная | Описание |
|------------|----------|
| `PUBLIC_WEB_ORIGIN` | HTTPS-адрес сайта **без** завершающего слэша, например `https://edify.su`. Из него собирается ссылка в письме. |
| `SMTP_HOST` | SMTP-сервер (у Reg.ru часто `smtp.reg.ru`). |
| `SMTP_PORT` | Порт: `465` (SSL) или `587` (STARTTLS). |
| `SMTP_SECURE` | `1` / `true` для SSL (465), `0` / `false` для STARTTLS (587). Если не задано, для порта 465 считается `true`. |
| `SMTP_USER` | Логин ящика. |
| `SMTP_PASS` | Пароль ящика (или пароль приложения, если провайдер так требует). |
| `MAIL_FROM` | Заголовок From, например `EDIFY <noreply@edify.su>`. |

Если что-то из списка не задано, эндпоинт запроса сброса отвечает **503** с понятным текстом (письма не уходят).

## Docker Compose

Переменные должны попасть **внутрь контейнера `api`**: задайте их в `.env` и убедитесь, что в `services.api.environment` вашего compose перечислены `PUBLIC_WEB_ORIGIN`, `SMTP_*`, `MAIL_FROM` (и при необходимости лимиты сброса). В репозитории корневой **`docker-compose.yml`** — это ссылка на **`docker-compose.prod.yml`**, где этот проброс уже есть; команда `docker compose --env-file .env up -d --build` подхватит его автоматически. Если у вас **свой** несвязанный `docker-compose.yml` без этих строк — скопируйте блок из `docker-compose.prod.yml` в сервис `api`, иначе API не увидит SMTP при любом `.env` на хосте.

## Reg.ru

1. Создайте почтовый ящик на домене (например `noreply@…`).
2. В справке Reg.ru возьмите хост/порт/тип шифрования SMTP.
3. Для доставляемости настройте **SPF** (и по возможности **DKIM**) для домена в DNS панели Reg.ru.
4. В справке часто указаны и **`smtp.reg.ru`**, и **`mail.hosting.reg.ru`**. Если один хост «висит» при подключении, попробуйте второй и проверьте пару порт + `SMTP_SECURE`.

## Лимиты

`PASSWORD_RESET_MAX_PER_EMAIL` (по умолчанию 5) и `PASSWORD_RESET_WINDOW_MS` (по умолчанию 3 600 000 мс = 1 час) ограничивают число запросов с одного ключа `IP:email` в памяти процесса API.

## 504 Gateway Time-out в браузере

Если в DevTools у **POST** `/auth/password/reset/request` статус **504**, nginx оборвал ожидание ответа от API: чаще всего **nodemailer долго ждёт SMTP** (неверный хост/порт, закрыт исходящий 465 на VPS, провайдер режет исходящую почту).

- С контейнера API: `docker compose exec api sh -lc 'nc -zv mail.hosting.reg.ru 465 || true'` (или ваш `SMTP_HOST` / порт).
- В конфиге **nginx** перед проксированием на API увеличьте таймауты, например: `proxy_connect_timeout 30s;`, `proxy_send_timeout 60s;`, `proxy_read_timeout 60s;` (или больше), и перезагрузите nginx.
- В API заданы разумные таймауты подключения к SMTP (~12–25 с), чтобы при недоступном сервере вернулась ошибка приложения, а не зависание до 504.
